# Supabase vs Notion — CS 대상자 관리 시스템 관점 비교

작성일: 2026-04-15
목적: API활용 CS 대상자 선정·관리 시스템에서 Notion을 어디까지 쓸지 결정하기 위한 판단 자료.

---

## 1. 한 줄 요약

Supabase는 "데이터·로직 엔진"이고, Notion은 "사람이 보고 손대는 운영판"이다. 둘은 대체재가 아니라 역할이 다르다. 기존 시스템(Phase 1~5 완료, 15개 테이블·5단계 자동심사·발송큐·모니터링뷰)을 버리고 Notion 단독으로 가는 선택지는 권장하기 어렵다. 그 이유와 비교 항목은 아래에 정리했다.

---

## 2. 작업 유형별 비교

| 작업 | Supabase | Notion | 판단 |
|------|----------|--------|------|
| 대량 데이터 적재(BRIS 6개월 이력 수만 건) | 매우 강함 (insert/upsert, 트리거) | 약함 (API rate limit, 페이지 단위) | Supabase |
| 5단계 자동 심사 SQL 함수 | 강함 (PL/pgSQL, 트랜잭션) | 불가 (Notion 자체 로직 없음, 외부 스크립트 필요) | Supabase |
| 6개월 발송 이력 필터링 | 강함 (인덱스+JOIN) | 가능하지만 느림 (최대 100건 페이지네이션) | Supabase |
| 통계·뷰(`v_cs_dashboard` 등) | 강함 (Materialized View 가능) | 약함 (수동 롤업/공식) | Supabase |
| 발송 토큰 발급, 응답 적재 API | 강함 (RPC, RLS) | 약함 (Webhook 처리에 별도 서버 필요) | Supabase |
| 담당자가 직접 보고 코멘트 | 약함 (별도 UI 필요) | **매우 강함** | Notion |
| 상태 변경(드롭다운 클릭) | 약함 (UI 직접 만들어야) | **매우 강함** | Notion |
| 사후관리 체크(기프티콘 발송여부 등) | 가능하지만 UI 필요 | **매우 강함** (체크박스/속성) | Notion |
| 다인 협업·코멘트·@멘션 | 약함 | **매우 강함** | Notion |
| 권한·공유 (외부 관리자에게 보기 권한) | 가능하지만 RLS 설정 필요 | 강함 (페이지 단위 공유) | Notion |
| 모바일 접근(현장에서 확인) | 직접 만들어야 | 강함 (앱 기본 제공) | Notion |
| 자체 설문 페이지에서 응답 적재 | 강함 (RPC, 토큰 검증) | 약함 (Form은 가능하나 토큰 검증 어려움) | Supabase |

## 3. 사용자가 원하는 작업 매핑

요청 사항: **발송 상태 + 응답여부 + 사후관리(기프티콘 발송여부, 6개월 이력 필터링 활용)**

| 요청 작업 | 어디서 처리하는 게 자연스러운가 |
|-----------|----------------------------------|
| 발송 상태 확인/변경 | Notion (담당자가 보고 클릭) |
| 응답여부 확인 | Notion 표시 + Supabase가 자동 채움 |
| 기프티콘 발송여부 체크 | **Notion** (수기 체크박스) |
| 6개월 이력 필터링 | **Supabase** (이미 `fn_cs_step4_check_survey_history` 존재). 결과만 Notion에 표시 |
| 1차 수동 발송 → 점차 외부 서비스 | Notion에 발송 대상 표시, 담당자가 외부 메일 클라이언트로 발송. 나중에 SendGrid 등을 Supabase Edge Function에 연결 |
| 자체 설문 페이지 응답 | Supabase에 적재 → Notion에 자동 동기화 |

## 4. 권장 아키텍처 (안)

**Supabase = 원본(source of truth), Notion = 운영판**

```
┌─────────────────────────────────────────────────────────┐
│ BRIS                                                    │
└──────────┬──────────────────────────────────────────────┘
           │ bris_to_supabase.py (cron, 매주 월 09:00)
           ▼
┌─────────────────────────────────────────────────────────┐
│ Supabase                                                │
│  - cs_target_batches → fn_cs_run_auto_screening         │
│  - 5단계 심사 결과 → cs_survey_targets (status=passed)  │
│  - 6개월 이력 검증, 토큰 발급                           │
└──────────┬──────────────────────────────────────────────┘
           │ Edge Function: notion_push (passed 건만)
           ▼
┌─────────────────────────────────────────────────────────┐
│ Notion DB: "CS 대상자 관리"                             │
│  속성:                                                  │
│   - 이름 / 회사 / 과정 / 차수 (BRIS 데이터, 읽기전용)   │
│   - 발송상태 (대기/발송완료/실패)  ← 담당자 변경       │
│   - 응답여부 (체크박스)            ← 자동/수동 모두    │
│   - 기프티콘 (체크박스)            ← 수기              │
│   - 비고 / 코멘트                  ← 협업              │
│   - 설문토큰 / 설문URL (읽기전용)                      │
└──────────┬──────────────────────────────────────────────┘
           │ Notion Webhook (속성 변경 감지)
           ▼
┌─────────────────────────────────────────────────────────┐
│ Supabase: 상태 역동기화                                 │
│  - 발송상태=발송완료 → cs_dispatch_records 기록         │
│  - 기프티콘=체크 → cs_survey_targets.gift_sent=true     │
└─────────────────────────────────────────────────────────┘

[자체 설문 페이지]
  설문 URL 응답 → Supabase RPC → cs_survey_participation
                              → Notion DB 응답여부 자동 체크
```

## 5. 불편한 질문 (결정 전에 답해야 할 것)

1. **데이터 권한** — Notion에 BRIS 고객 정보(이메일, 전화번호)를 옮기는 게 사내 보안 정책상 가능한가? 외부 SaaS에 개인정보 적재 가능 여부 확인 필요.
2. **양방향 동기화의 비용** — Notion ↔ Supabase 양방향은 충돌 처리(같은 행을 두 곳에서 동시에 수정)가 까다롭다. 단방향(Supabase → Notion)이면 운영자는 Notion에서 못 바꾼다. 어느 쪽이 운영 흐름에 맞나?
3. **사후관리 항목 추가 가능성** — 기프티콘 외에 추후 "재인터뷰 필요", "VOC 등급" 같은 항목이 늘어날 수 있나? 늘어난다면 Notion이 유연하다(스키마 변경 클릭 한 번). Supabase는 마이그레이션 필요.
4. **기존 대시보드 운명** — `cs_dashboard.html`은 계속 쓸 건가, 폐기인가? Notion으로 대체하면 동기화 코스트만 들고 대시보드 가치가 사라진다. 같이 쓰면 "어디서 봐야 하지?" 혼란 발생.
5. **Notion API 한도** — 무료/팀 플랜 모두 초당 3 req 정도 제한. 수백~수천 명 대상자를 한 번에 푸시할 때 배치·재시도 로직 필요.
6. **누가 운영하나** — 담당자 1명인가, 여러 명인가? 1명이면 Notion 도입의 협업 가치가 줄어든다.

---

## 6. 다음 단계 제안

위 권장 아키텍처(Supabase 원본 + Notion 운영판, 단방향 우선)로 진행할지, 아니면 다른 방향을 고를지 결정하면, 결정에 맞춰서 다음을 만든다:

- Notion DB 스키마 정의 (속성 목록 + 옵션값)
- Supabase Edge Function: `notion_push` (passed 건 → Notion 페이지 생성)
- Notion 변경 → Supabase 역동기화 (Webhook 또는 polling)
- 자체 설문 페이지 라우트 (`/survey/:token` → Supabase RPC)
- 대시보드 정리(폐기/유지/통합)
