# 09. PRD 프레임워크 — 문제 해결 문서로서의 PRD

> **핵심 원칙.** PRD 는 "만들 것의 쇼핑 리스트" 가 아니라 *문제 정의 → 가설 → 검증 → 결정* 의 궤적을 담는 Living Document 다. 기능 나열이 아니라 "왜" 를 전달한다.

## 현상 진단

- 현재 `docs/PRD.md` 단일 파일로 존재. 시간이 지나면서 섹션이 누적되어 *특정 기능의 현 상태* 를 찾기 어렵다.
- 신규 기능 기획이 Slack/Notion 문서와 혼재되어 있어 개발·QA 가 "진짜 요구사항" 을 물어야 한다.
- 엣지 케이스(로딩·권한·네트워크·한국어 IME·모바일)를 체계적으로 점검하는 체크리스트 부재.
- 추론 근거(귀납·연역·가추)가 기록되지 않아 시간이 지나면 "왜 이 방향?" 을 추적하기 어렵다.

## 설계 원칙

### A. 문서 구조

```
docs/prd/
├── README.md              # 인덱스: 현재 active PRD 목록 + 상태
├── templates/
│   ├── prd.md             # 기본 템플릿 (= _templates/prd.md 와 동일 소스)
│   └── rfc.md             # 작은 변경용 단일 페이지 RFC
├── active/
│   └── <YYYY-MM-DD>_<slug>.md   # 작업 중 PRD
└── archive/
    └── <YYYY-MM-DD>_<slug>.md   # 출시 완료·폐기 PRD
```

- 신규 PRD 는 항상 `active/` 에서 시작.
- 출시·폐기되면 `archive/` 로 이동하면서 맨 앞에 Outcome 섹션 추가 (목표 달성 여부, 수용지표 실제값).
- 기존 `docs/PRD.md` 는 상단에 "이 문서는 `docs/prd/active/` 로 이관되고 있어요" 배너 + 현 PRD 요약만 유지.

### B. 표준 섹션 (모든 PRD 필수)

1. **제품 개요 및 배경** — 문제 + 관측 데이터(VOC, 이탈률). 개인 짐작 금지.
2. **비즈니스 목표 및 성공 지표** — "결제 이탈률 15% 감소" 같은 정량 숫자.
3. **타겟 사용자 / 페르소나** — 뾰족하게 ("주 3회 모바일로 이력서 업데이트하는 IT 취준생").
4. **핵심 사용자 여정(CUJ)** / 사용자 스토리 — "(유형)은 (목적)을 위해 (기능)이 필요하다" 템플릿.
5. **기능 요구사항** — MoSCoW 로 구분 (Must/Should/Could/Won't).
6. **엣지 케이스 및 예외 처리** — §D 체크리스트 기반.
7. **개발·배포 타임라인** — 종속성·병목 문서화.
8. **참조 리소스** — Figma/스키마/리서치 링크 (이미지 임베드 지양, 링크만).

### C. 추론 모드

신규 PRD 의 "배경·가설" 단락에서 **귀납·연역·가추** 중 어느 방식으로 결론에 이르렀는지 명시한다.

- **귀납 (Induction)**: 여러 관찰 → 일반화.
  - 예: *"지난 30일 `/s` 경로 이탈의 62% 가 3번째 섹션 직후 발생했고(로그 N=…), 3번째 섹션 평균 문항 수는 12개다. → 가설: 섹션당 문항 수가 이탈의 주원인이다."*
- **연역 (Deduction)**: 일반 전제 → 특정 결론.
  - 예: *"사용자는 편익이 불분명하면 이탈한다 (UX 일반 원칙). 우리의 현 랜딩은 추상 카피뿐이다. ∴ 편익을 상단에 노출해야 한다."*
- **가추 (Abduction)**: 관찰 → 가장 그럴듯한 설명 → 검증 대상 가설.
  - 예: *"금요일 저녁 20대 응답률이 급감한다. 야외 셀룰러에서 로딩 지연이 원인일 수 있다 (가설). A/B 로 경량 번들을 실험한다."*

가설은 그대로 요구사항이 아니다. 반드시 **검증 방법**(A/B, UT, 계측)을 명시한다.

### D. 엣지 케이스 체크리스트 (필수 통과)

- [ ] 네트워크: 지연(>3s)·오프라인·간헐 끊김 시 복구 동선.
- [ ] 권한: 비로그인·세션 만료·권한 없음·차단.
- [ ] 빈 데이터: 처음 접속·결과 0건·삭제 직후.
- [ ] 한국어 IME: 조합형 입력·복사·붙여넣기 시 포커스 유지.
- [ ] 모바일: iOS Safari 바텀 바 가림·홈 인디케이터·키보드 영역·세이프 에어리어.
- [ ] 동시성: 같은 링크 2창에서 편집·중복 제출·낙관 UI 실패.
- [ ] 브라우저: 뒤로가기·새로고침·북마크·딥링크 재진입.
- [ ] 시간: TZ(KST)·만료·쿨다운·캐시 무효화.
- [ ] 다국어 대비: 카피 길이 편차·줄바꿈 허용.
- [ ] AI: 미설정 키·rate limit·타임아웃·결과 품질 저하.

### E. Living Document 규칙

- PR 머지 시 해당 PRD 의 "변경 이력" 에 `YYYY-MM-DD | 변경 내용 | PR #` 추가.
- 출시 후 30일 내 "Outcome" 섹션 작성 (수용지표 실제값).
- Outcome 이후 움직임은 신규 PRD 로 분기.

### F. UX 요구사항의 언어화

모호한 표현(`세련된 디자인`, `직관적`) 금지. 대신 시스템의 작동 원리를 구체 기술.

```
❌ "취소와 승인 버튼을 잘 구분한다."
✅ "취소 버튼은 outline variant, 승인 버튼은 default variant (brand primary 솔리드).
   우측 하단 고정, 간격 8px, 엄지영역 안."
```

로딩 상태는 낙관적 UI / 스켈레톤 / 진행바 중 무엇을 쓰는지 명시 (`04-respondent-ux.md` §E·D 참고).

## 사용 흐름

1. 새 기능이 발화되면 `_templates/prd.md` 복사 → `docs/prd/active/<YYYY-MM-DD>_<slug>.md`.
2. §C 추론 모드 최소 1가지 명시, §D 체크리스트 관통.
3. 팀 리뷰(디자인·엔지니어·QA·법무·데이터) → MoSCoW 조율 → 승인.
4. 개발 중 변경은 PRD 에 반영 후 PR. 구두 합의 금지.
5. 출시 후 Outcome 작성 → `archive/` 이동.

## 수용지표

**정량**
- `docs/prd/active/` 에 신규 PRD ≥ 3건 (Phase 2 말).
- PR 본문에 PRD 링크 첨부율 ≥ 80%.
- 엣지 케이스 §D 체크리스트 미통과로 반려된 PRD 비율 측정 가능(점검 존재 자체가 지표).

**정성**
- "왜 이 기능이 있지?" 질문이 PRD 링크 1개로 해결된다.
- 신규 팀원이 최근 archive PRD 2~3개만 읽고 제품 방향을 이해한다.

## 체크리스트 (PRD 작성자)

- [ ] 배경에 정량 근거(지표·인용)를 포함했다.
- [ ] 추론 모드(귀납·연역·가추)를 명시했다.
- [ ] 성공 지표가 측정 가능한 숫자다.
- [ ] 페르소나가 뾰족하다.
- [ ] CUJ 가 스토리 템플릿을 따른다.
- [ ] MoSCoW 로 기능 우선순위를 구분했다.
- [ ] §D 엣지 케이스를 관통했다.
- [ ] 모호한 UX 표현을 시스템 언어로 바꿨다.
- [ ] 검증 방법(A/B·UT·계측)을 적었다.
- [ ] 참조 리소스가 링크다(이미지 임베드 지양).
